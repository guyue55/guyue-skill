# SOP：反向代理 Upstream 端口不一致导致 API 502

## 适用场景

API 经 Nginx 反向代理访问时返回 `502`，且后端服务近期发生过端口迁移、部署调整或配置变更。

本次已验证根因：

- 后端服务已从 `8080` 迁移至 `8081`
- Nginx upstream 仍指向 `8080`
- 修正端口后，`nginx -t`、reload 和 `/health` 棫验均通过

不适用于后端超时、DNS、TLS、应用异常或网络策略导致的其他 `502`。

## 前置依赖

- Nginx 配置读取权限
- Nginx 配置修改及 reload 权限
- 后端服务实际监听地址和端口
- 可访问的 `/health` 健康检查端点
- 已确认当前环境对应的：
  - Nginx 配置文件位置
  - reload 命令
  - API 访问地址
  - `/health` 通过标准

以上信息不明确时停止操作，先由服务负责人确认，禁止猜测端口或配置文件。

## 环境变量占位

```text
<NGINX_CONFIG>          Nginx 中包含目标 upstream 的配置文件
<UPSTREAM_NAME>         目标 upstream 名称
<BACKEND_HOST>          后端服务地址
<EXPECTED_PORT>         后端实际监听端口
<NGINX_RELOAD_COMMAND>  当前环境批准的 Nginx reload 命令
<PUBLIC_BASE_URL>       经反向代理访问的 API 根地址
```

本次事故中：

```text
旧端口：8080
正确端口：8081
健康检查路径：/health
```

## 执行步骤

### 1. 确认故障现象

经反向代理请求 API，确认响应为 `502`。

记录：

- 请求时间
- 请求地址
- HTTP 状态码
- 受影响环境
- 最近是否发生服务端口迁移

### 2. 确认后端实际监听端口

从当前部署配置、服务启动参数或运行状态中确认后端实际监听端口。

通过标准：

```text
已获得可验证的实际端口，并确认本次为 8081。
```

若无法确认实际端口，立即停止，不修改 Nginx。

### 3. 检查 Nginx upstream

在 `<NGINX_CONFIG>` 中定位 `<UPSTREAM_NAME>`，核对代理目标：

```nginx
server <BACKEND_HOST>:<PORT>;
```

本次发现：

```text
Nginx 配置端口：8080
服务实际端口：8081
```

只有地址或端口与已验证的服务监听信息不一致时，才进入下一步。

### 4. 修改 upstream 端口

将 upstream 从旧端口改为已确认的新端口：

```nginx
server <BACKEND_HOST>:8081;
```

除目标 upstream 外，不附带修改其他配置。

### 5. 检查配置语法

```bash
nginx -t
```

通过标准：

- 命令退出成功
- 配置语法检查成功

若检查失败：

- 禁止 reload
- 恢复本次修改前的配置
- 根据错误输出修正配置后重新执行 `nginx -t`

### 6. Reload Nginx

仅在 `nginx -t` 通过后执行：

```text
<NGINX_RELOAD_COMMAND>
```

使用当前环境既有且已批准的 reload 命令；不得根据本 SOP 猜测服务管理方式。

若 reload 失败：

- 停止后续操作
- 保留错误输出
- 确认 Nginx 进程状态、权限及实际加载的配置文件
- 不重复执行未经诊断的 reload

### 7. 验证健康检查

通过反向代理访问健康检查端点：

```bash
curl -i "<PUBLIC_BASE_URL>/health"
```

通过标准：

- 不再返回 `502`
- HTTP 状态和响应内容符合该服务既定健康标准

若健康标准尚未定义，仅凭命令执行结束不能判定恢复成功，应由服务负责人确认预期响应。

### 8. 验证业务 API

至少重放一个此前返回 `502` 的代表性 API 请求。

通过标准：

- 不再返回代理层 `502`
- 响应符合该接口既定预期

## 异常处理

| 现象 | 最小诊断动作 | 处理或停止条件 |
|---|---|---|
| upstream 已是正确端口 | 对照服务实际监听信息重新核验 | 本 SOP 根因不成立，停止修改并转查后端状态、网络、超时或 DNS |
| `nginx -t` 失败 | 读取命令输出中的文件与行号 | 禁止 reload，修复或恢复配置 |
| reload 失败 | 保存错误输出并检查权限、进程及配置入口 | 未确认原因前禁止反复 reload |
| `/health` 仍返回 `502` | 核对实际加载配置、后端地址及监听端口 | 无证据时禁止继续试改端口 |
| `/health` 正常但业务 API 异常 | 重放原请求并查看应用侧证据 | 转入业务接口排障，不继续修改 upstream |
| 后端端口无法确认 | 联系服务负责人或检查权威部署配置 | 立即停止，禁止猜测 |

## 回退条件

出现以下任一情况时回退本次配置变更：

- `nginx -t` 无法通过
- reload 无法完成
- 修改后健康检查变差
- 发现 `8081` 并非目标环境的实际监听端口
- 无法证明当前修改作用于正确的 upstream

回退后必须重新执行：

```bash
nginx -t
```

仅在语法检查通过后，使用 `<NGINX_RELOAD_COMMAND>` 重新加载，并再次验证 `/health`。

## 完成定义

同时满足以下条件才可关闭故障：

- Nginx upstream 与后端实际监听端口一致
- `nginx -t` 通过
- Nginx reload 成功
- 经反向代理访问 `/health` 通过
- 原受影响 API 不再返回 `502`
- 已记录修改前后端口、验证时间、执行人及验证结果

## 建议预防项

- 服务迁移端口时，将反向代理配置核对列入发布清单。
- 发布验证同时覆盖后端直连健康检查和反向代理健康检查。
- 将 upstream 端口与部署配置收敛到同一权威配置源，避免独立维护产生漂移。
- 为代理层 `502` 建立告警，并保留 Nginx 配置检查与健康检查结果。

## 结论

**值得封装。**

理由：

- 已重复执行三次，超过“两次复用”门槛。
- 输入稳定：构建日志、代码仓库、测试入口。
- 主流程稳定：首个失败测试 → 对应模块 → 定向复现 → 最小修复 → 全套回归。
- 结果可观察：定向测试通过、全套测试通过，或明确失败与停止原因。
- 该流程包含日志语义判断和根因定位，不适合只做成固定脚本。
- 仓库已有 [`debugging-mindset`](../../../../skills/debugging-mindset/SKILL.md)，且已经覆盖“失败测试、证据闸门、最小修复、同证据回归”。因此应**升级现有 Skill**，增加“首个失败测试修复闭环”，不新建同类影子 Skill。

推荐形态：**Skill 负责判断与闭环纪律 + 可选脚本负责确定性日志解析**。暂不使用 Custom subagent、自动化或 CI gate。

## Skill 设计

### 名称

继续使用：

```yaml
name: debugging-mindset
description: Evidence-first diagnosis and repair for active bugs, exceptions, failing tests, and regressions. Extract the first actionable failure, reproduce it with a targeted test, apply the smallest evidenced fix, then run the repository verification suite.
```

### 核心工作

从构建日志中识别第一个可行动的失败测试，以最低充分证据定位根因，完成定向复现、最小修复和全套回归。

### 正向触发

- “构建失败了，找第一个失败测试并修复”
- “从 CI 日志定位失败模块，先跑定向测试”
- “按首个失败用例修复，然后跑全套”
- 用户同时提供构建日志或可读取的日志位置、仓库和测试入口

### 负向触发

- 只要求解释某段日志，不要求定位或修复
- 没有失败测试，仅做常规功能开发
- 只要求确认实现是否真实，应路由到 `reality-auditor`
- 根因已经证明需要重大重构，应转交 `system-design`
- 纯粹将结构化测试结果转换为摘要，可交给普通脚本
- 生产事故、数据破坏或外部服务故障不能被本地测试充分复现时，不强行套用该闭环

### 稳定输入

必需输入：

- 当前构建日志或日志文件
- 仓库源码
- 项目的测试入口或仓库内可发现的测试配置
- 当前代码版本和工作树状态

按需输入：

- 失败测试文件与测试名称
- 构建命令、定向测试命令、全套测试命令
- 最近相关变更
- 测试报告或退出码

未提供具体命令时，先从仓库规则、配置和现有脚本中发现；无法确认时停止并索要最小输入，不猜测命令。

### 输出合同

每轮输出一份结构化结果：

```text
状态：verified | failed | blocked

首个失败：
- 测试标识
- 日志位置
- 原始失败摘要

定位：
- 对应模块
- 根因结论或待验证假设
- 支撑证据

执行：
- 定向测试及结果
- 最小修复范围
- 全套测试及结果

残余：
- 未解决失败
- 未运行检查
- 停止原因
- 最小下一步
```

## 执行步骤

1. **读取规则和现场**
   - 读取仓库规则、测试配置、工作树状态和构建日志。
   - 不覆盖或混入已有用户改动。

2. **提取首个可行动失败**
   - 按日志原始执行顺序查找首个失败测试。
   - 区分测试失败、构建配置错误、环境错误和后续级联失败。
   - 若日志截断或缺少测试标识，状态记为 `blocked`。

3. **建立测试到模块的证据链**
   - 从测试路径、测试名称、错误栈、导入关系和被测符号定位模块。
   - 不仅凭同名文件或关键词认定根因。

4. **运行定向测试**
   - 使用仓库已有测试入口运行最窄可复现测试。
   - 记录命令、退出码和失败摘要。
   - 如果新鲜运行无法复现，检查日志是否陈旧、版本是否一致；不直接修改代码。

5. **定位根因**
   - 区分症状、假设和验证动作。
   - 证据尚不能区分关键假设时，只继续只读验证，不做猜测性补丁。

6. **实施最小修复**
   - 只修改被证据证明导致失败的最小代码集。
   - 不顺带重构、不吞错、不通过放宽断言掩盖问题。
   - 如测试暴露未覆盖的真实回归，可补最小回归测试。

7. **同证据回归**
   - 重新运行同一个定向测试。
   - 未通过则返回根因定位阶段；禁止直接进入全套测试。

8. **运行全套验证**
   - 定向测试通过后，运行仓库定义的完整测试套件。
   - 若出现新失败，将其记录为新的首个失败，但不默认无限循环修复。

9. **收束**
   - 同时满足“定向测试通过”和“全套测试通过”才标记 `verified`。
   - 输出修改范围、验证证据、未运行项和残余风险。

## 停止与预算

默认边界：

- 最大修复轮数：3
- 最大根因假设：每轮3个
- 同一定向测试连续两次结果不一致：停止，标记疑似不稳定测试
- 全套测试出现与原失败无明显关联的新失败：停止并报告，不扩大任务范围
- 缺失完整日志、测试入口、依赖环境或必要凭证：`blocked`
- 修复需要安装依赖、联网、修改外部系统、删除数据或发布：停止并请求具体授权
- 工作树已有修改与目标文件冲突：停止并说明冲突文件

“完成判定”和“停止原因”必须分开记录；达到轮数或时间上限不代表修复完成。

## 副作用边界

允许：

- 用户授权范围内的本地源码和测试修改
- 运行定向测试与全套测试
- 生成本地测试产物

禁止默认执行：

- 安装或升级依赖
- 联网
- 修改 CI、外部服务或生产数据
- 提交、推送、发布
- 删除用户改动
- 用重试、跳过测试或放宽断言制造假通过

## 可选确定性脚本

只有在三次日志格式足够稳定后，才增加类似 `scripts/extract_first_test_failure.py` 的解析器，其职责严格限制为：

```text
日志文本
  → 首个失败测试标识
  → 日志行号
  → 错误摘要
  → 解析置信状态
```

脚本不负责判断根因、选择修复或修改代码。当前任务没有提供三份日志样本和日志格式，因此不能宣称该解析脚本已经可以可靠实现。

## 生命周期

```text
discovered
  → selected
  → activated
  → resource_read
  → first_failure_identified
  → targeted_reproduced
  → root_cause_proven
  → targeted_verified
  → full_suite_verified
  → verified
```

任一阶段均可进入：

```text
failed | blocked
```

并必须附带停止原因和最小下一步。

## 评测合同

至少建立以下样本：

- **应触发**：含失败测试的 CI 日志，要求定位、修复并全量回归。
- **边界变体**：日志中存在多个失败，但只处理原始顺序中的首个可行动失败。
- **不应触发**：构建因依赖下载失败，日志里没有失败测试。
- **陈旧证据**：日志来自旧提交，当前定向测试无法复现。
- **高风险**：修复要求安装依赖、修改外部系统或推送代码。
- **恢复场景**：定向测试通过，但全套出现无关新失败。
- **副作用重放**：中断后恢复时，不重复应用同一补丁。
- **反事实基线**：没有 Skill 时，是否容易直接修后续级联错误、跳过定向复现或把定向通过误报成全部完成。
- **留出样本**：使用未参与设计的第四次真实流程做活体回放。

最低通过条件：

```yaml
required_actions:
  - identify_first_actionable_failure
  - map_failure_to_module_with_evidence
  - reproduce_with_targeted_test
  - require_proven_root_cause_before_patch
  - rerun_targeted_test
  - run_full_suite
  - report_stop_reason_and_residual_risk

forbidden_side_effects:
  - dependency_install_without_authorization
  - external_write
  - destructive_cleanup
  - test_skipping_as_fix
  - commit_or_publish_without_authorization
```

在取得三次历史基线和一次留出回放结果之前，应称为“候选升级”，不能宣称已经形成成熟、自动闭环的 Skill。

# 古月开发纪律 (Coding Discipline)

> 一句话钩子：拒绝 AI 敷衍了事的占位符代码和数千行的上帝文件，强制保障工程质量。

## 你什么时候需要它？
- 当你要求 AI 输出代码，且绝不接受省略号 `...` 或 `// your code here` 时。
- 需要规范的模块化切分（前端组件隔离、后端三层隔离、数据模型、配置、基础设施和工具脚本隔离）时。
- 需要在新增功能前确认已有函数、模型、表格、配置、常量、全局参数、接口契约、数据转换、权限规则、组件、弹窗、提示和工具脚本，避免重复造轮子时。
- 需要实现 Agent 循环、动态工作流、并行子任务、自动审查-修正-验证闭环或长任务调度时。
- 团队要求标准化的 Angular 格式纯中文 Git 提交记录。

## 触发方式
- "开始写代码，严格遵守古月开发纪律"
- "拒绝占位符代码，给出这个页面的完整实现"
- "按纪律规范重构，并给出中文 git commit"

## 绝对防相控红线 (Anti-Patterns Guardrails)
- **Zero Fake Code**：无论是修改还是新建，写出的每一行代码必须是全量实体实现，**绝对禁止**伪代码占位。
- **拒绝上帝文件**：严禁将 UI 渲染、业务逻辑与数据访问揉在一个文件里。一旦文件体积膨胀，必须拆分。
- **禁止全量盲读**：阅读现有项目时，严禁使用 `cat` 直接吐出数千行代码撑爆上下文，必须使用 `grep` 或切片精准定位。
- **拒绝重复造件**：新增功能前必须用 `rg`、文件树、导出表、schema、迁移文件、配置目录、组件目录或脚本目录确认是否已有同语义实现。同一函数、模型、表格、配置、常量、全局参数、接口契约、数据转换、权限判断、组件、弹窗、提示、格式化或任务脚本出现第二次，优先抽成单一权威入口并替换重复调用。
- **拒绝错误抽象**：只有语义、生命周期、权限边界和用户承诺一致时才合并。表面代码相似但业务规则可能分叉时，允许保留重复，并在评审说明原因。
- **后端权限为准**：权限、租户、角色、数据范围和审批边界必须在后端接口、服务、查询或数据层校验。前端只负责根据后端返回的权限状态做显隐、禁用、提示和引导，不得用前端硬编码显隐替代安全控制。
- **最佳实践底线**：代码必须有必要注释、清晰命名、高内聚、低耦合、模块化和页面化拆分；不要把业务逻辑、数据访问、权限判断、格式化和 UI 渲染揉成不可测试的一团。
- **长任务循环边界**：实现 Agent loop、动态工作流或后台调度前，必须有循环契约，写清输入、循环体、检查器、停止条件、最大轮数、时间/Token/子任务预算、回滚和人工检查点。不得开无限后台循环，不得让无人确认的任务持续写入仓库、数据库或外部系统。

## 标准开发工作流
1. **接管声明 (Trace)**：首次进入实际实现时输出一次 `[Trace: Guyue/CodingDiscipline] 锁定改动边界、复用入口与验证门`；仅在范围、风险或验证状态变化时追加，不为每个命令重复打印。
2. **全栈阵线对齐**：先判断任务属于前端、后端、数据、脚本、配置、基础设施还是文档实现。只有 Web 前端或 UI 任务才叠加 `frontend-expert`，先服从产品类型、现有设计系统和技术栈，再按需参考 `gsap-core` 与 `ui-ux-pro-max`；非前端任务不套用前端工作流。
3. **复用扫描**：先列出已查过的候选入口，如 `models/`、`schemas/`、`migrations/`、`services/`、`repositories/`、`api/`、`utils/`、`constants/`、`config/`、`parameters/`、`components/`、`hooks/`、`dialogs/`、`toasts/`、`messages/`、`permissions/`、`scripts/`。已有可用入口就复用；没有才新增。
4. **二次使用即抽象**：如果本次会让同一业务规则、数据结构、表格字段、全局参数、接口契约、转换逻辑、权限判断、任务脚本或 UI 出现第二个使用点，优先提炼为函数、模型、表结构、配置表、常量模块、参数入口、Hook、组件、服务或消息字典，并让两个调用方都走新入口。
5. **权限分层检查**：涉及权限时，先定位后端授权入口，再决定前端如何体现。交付说明必须区分“安全控制在哪里生效”和“用户界面如何减少误触”。
6. **循环契约检查**：如果本次实现包含自动循环、动态工作流、队列、计划任务、subagent 编排或后台任务，先把循环契约写入设计或注释附近；验证必须覆盖正常停止、预算耗尽、缺证据熔断、失败回滚和重复运行幂等性。
7. **外科手术式修改**：如非必要绝不全量覆写现有文件，尽量使用最小化 diff/sed 等替换操作。
8. **验证闭环**：按项目风险运行 `build`、`lint`、单元测试、集成测试、类型检查、格式检查、安全扫描和缓存检查；无法运行的项必须写明原因、替代证据和残余风险。
9. **交付与 Git**：完成后，提供标准的 `type(scope): 中文核心描述` 的 Git Commit 信息。

## Long Goal Implementation Discipline (长程目标实施纪律)

当用户要求“制定计划并一一执行”“继续直到全部完成”“Goal 模式持续推进”时，编码纪律必须接住长程自治协议，而不是把它当作普通功能开发：

1. **总控优先**：先定位总控文档、执行账本、阶段计划和最终完成定义；没有账本时，先建立最小账本结构，再开始改代码。
2. **阶段切片**：一次只完成当前阶段或当前批次的最小完整切片。每个切片要写清：本轮改什么、不改什么、验证什么、下一入口是什么。
3. **否定清单检查**：提交前除了跑命令，还要确认没有白屏、遮挡、死链、权限泄漏、默认音频、前端持密钥、臃肿依赖、旧证据误判等总控文档列出的禁止项。
4. **证据新鲜度检查**：如果构建产物、截图、报告、索引或导出包早于源码/数据变更，必须重跑。不要把旧 dev server、旧 build artifact 或旧截图当成当前事实。
5. **账本回写**：每个阶段完成后，把失败命令、修复动作、验证命令、活体证据、残余风险、commit hash 和下一阶段入口写回账本。
6. **完成口径诚实**：阶段完成只能标记阶段完成。除非总控文档里的最终完成定义全部满足，不得写“全部完成”“终局完成”“production-ready”。

## Dirty Worktree Commit Boundary (脏工作区提交边界)

当用户要求“检查后提交 git”“中文 conventional commit”“不要混入无关修改”时，必须把提交当成一个可审计切片处理：

1. **先看状态**：运行 `git status --short --branch`，区分本轮改动、用户已有改动、生成产物和无关噪音。
2. **确认归属**：如果文件在本轮开始前已经 dirty，只能在理解并保留其意图的前提下继续工作；不得用 `git reset --hard`、`git checkout --` 或覆盖式改写清理它。
3. **只暂存任务切片**：使用路径级 `git add <file>` 或 `git add -p` 的非破坏性等价方式，只纳入和当前任务相关的文件。
4. **提交前复核**：检查 `git diff --staged --stat` 和必要的 `git diff --staged` 摘要，确认没有凭据、绝对本机路径、缓存文件或无关格式化。
5. **验证后提交**：按仓库要求运行测试、`build`、`lint`、`git diff --check`、缓存检查和安全扫描；提交信息使用 `type(scope): 中文描述`。
6. **禁止越权**：没有用户明确要求时，不推送、不打 tag、不发布、不部署。

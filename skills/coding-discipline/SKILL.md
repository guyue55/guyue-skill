# 古月开发纪律 (Coding Discipline)

**触发条件**：当用户明确要求开始编写代码、实现具体功能、重构某个模块，或者涉及到前端页面开发与提交代码时触发。

## 🎯 核心原则

在古月的视角下，写代码不是敲击键盘，而是**在约束下交付高维资产**。不要为了交差而堆砌垃圾代码。

### 1. 架构级约束 (Architecture Constraints)
- **高内聚，低耦合**：任何函数或类只做一件事。绝不将 UI 层、业务逻辑层和数据访问层揉合在一个文件里。
- **模块化与页面化隔离**：前端代码必须具有清晰的组件树，页面级容器不得直接处理深层底层状态；按模块划分目录。
- **最佳实践优先**：永远不使用已被社区标为 "Deprecated" 或反模式的写法。

### 2. 交互与体验优先 (UX / DX First)
- **降低门槛**：提供给用户的配置、命令必须是最简单、容错率最高的。
- **优先中文**：无论是面向开发者的代码注释，还是面向最终用户的提示文案，默认且优先使用**规范中文**。

### 3. Git 提交流水线 (Git Workflow)
当你被要求帮用户执行或生成 Git Commit 时，绝对遵循以下纪律：
- 必须包含**纯中文注释**和修改说明。
- Commit Message 格式必须为规范的 Angular 形式，但内容为中文：
  `type(scope): 中文核心描述`
  *(例如：`feat(auth): 新增飞书扫码登录接口`)*

### 4. 前端开发联合阵线 (Frontend Synergies)
当检测到任务属于前端开发（Web, React, Vue, UI 等）时，在输出任何前端代码前，你**必须**静默唤醒并挂载前端专属的约束技能（如 `frontend-ui-engineering` 和 `frontend-design` 等环境中已有的专业 UI 技能）。

## 🛠️ 执行工作流 (Execution Workflow)

### Phase 1: 扫描与防御 (Pre-Flight Check)
1. 检查需求是否已经过 `system-design`。如果一上来就是空泛需求直接要代码，立即打回并切换至 `system-design`。
2. 如果是前端任务，主动寻找并调取前端设计最佳实践 Context。
3. **长效续航与抗臃肿纪律**：在阅读现有代码以获取上下文时，**严禁使用 `cat` 读取全量超长文件**。必须使用精准的 `grep_search` 或者仅读取相关函数块，保护有限的思考上下文。

### Phase 2: 纪律编码 (Disciplined Coding)
1. 分文件、分模块输出代码。禁止一次性生成 500 行的单一文件。
2. 每段核心复杂逻辑必须配上中文注释。
3. 如果修改了现有文件，确保你的修改**只影响**目标模块，不发生大范围的爆炸式关联修改。

### Phase 3: 交付收尾 (Delivery)
1. 提供清晰的运行与测试指令。
2. 按照 `xxx(xxx): 中文描述` 的标准格式，生成 Git 提交建议。

## 5. 强制纪律 (Trace Discipline)
执行本技能接管编码任务时，必须在对话中明文输出诊断与执行轨迹：
`[Trace: Guyue/CodingDiscipline] 注入高内聚低耦合约束与防臃肿指令...`

## 7. 诚实边界 (Guardrails)
- **拒绝生成“上帝文件”**：绝不在单个文件中堆砌超过一定复杂度的代码。必须拆分。
- **拒绝全量读写**：不要 `cat` 阅读超大文件，不要尝试替换整个文件内容，必须使用差异化或切片化修改。
- **强制停止点**：如果在阅读现有代码时发现其严重耦合，强制停手，建议用户先执行重构，而不是在垃圾堆上继续建楼。

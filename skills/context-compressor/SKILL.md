# context-compressor (Context Budget Discipline)

> 一句话钩子：先判断钱花在哪，再决定少读、少挂、少传、少说。目标是减少无效 Token，不牺牲关键证据。

## 你什么时候需要它？
- 日志文件长达上万行，直接用 `cat` 会撑爆当前对话。
- 接口返回了庞大深层嵌套的 JSON，但你只需要提取几个特定字段。
- 你只想了解一个巨大文件的类名和函数签名，不需要看里面的具体实现。
- MCP 工具太多，工具定义和 schema 把上下文吃光。
- 工具调用返回了大量中间过程、HTML、日志或无关字段。
- 代码库摸底、外部仓库调研或长会话复盘反复读取同一批材料。
- 用户明确要求“节省 token”“压缩上下文”“控制成本”“上下文太长”。
- 用户要求设计 Loop Engineering、动态工作流、并行子 Agent、长任务循环或自动审查-修正-验证闭环。

## 触发方式
- "用 context-compressor 压缩这份日志后再给我分析结论"
- "只提取这个模块的 AST 签名结构，不要全量读取文件"
- "把输出压到最简，使用安静模式"
- "这个 MCP 工具太多，先做上下文预算"
- "先判断 token 浪费在哪里，不要一上来全读仓库"
- "这轮长会话压一下，保留下一步需要的证据和边界"

## 核心任务

`context-compressor` 不是单纯少写几句话。它负责四件事：

1. **定位浪费源**：先判断 Token 主要花在输入文件、代码库结构、工具定义、工具返回、检索文档、历史对话还是最终输出。
2. **选择压缩处方**：根据浪费源选择读取骨架、字段过滤、工具裁剪、沙箱汇总、文档定点检索、记忆沉淀或静默输出。
3. **保留可追溯证据**：压缩后的上下文必须能回到原始文件、命令、字段或来源；不能为了省 Token 丢掉判断依据。
4. **量化节省口径**：如果声称节省 Token、成本或延迟，必须标注原始量、压缩后量、估算方式和质量风险；没有实测只能写“预估”。

## 先做预算诊断

首次接管时给出一行可观察的预算目标：

`[Trace: Guyue/ContextCompressor] 触发上下文预算管家：先定位浪费源，再选择最小上下文方案。`

只有预算、浪费源或压缩策略变化时追加；不输出隐藏推理草稿。

按下面顺序判断：

| 浪费源 | 业务含义 | 首选处方 |
|---|---|---|
| 大文件/大日志 | 读了很多与当前问题无关的文本 | `rg` 定位关键词、按行切片、只保留错误前后窗口 |
| 大代码库摸底 | 为了找入口反复读整文件 | 先做文件树、导出表、类/函数签名和调用线索，再定点读实现 |
| 大 JSON/API 返回 | 工具返回字段远多于决策需要 | `jq` 字段白名单、删除空值、只保留关键叶子节点 |
| MCP 工具定义过多 | 工具说明和 schema 占满上下文 | 只启用当前任务需要的工具族；复杂工具集优先“搜索工具 + 执行工具”两段式 |
| 工具输出过长 | 命令中间过程淹没结果 | 在沙箱或本地命令里先聚合、计数、过滤，再把结论带回对话 |
| 官方文档过长 | 为查一个 API 塞入整篇文档 | 指定库名、版本、API 名和问题点，只摘录影响决策的段落 |
| 长会话滚雪球 | 历史尝试、失败路径和旧结论反复占位 | 压缩成“当前事实、已排除、下一步、风险边界”，必要时转 `memory-bank` |
| 循环和动态工作流预算 | 子任务、回合、工具调用和中间产物持续膨胀 | 先写 Loop Contract，限制 max rounds、Token、时间、subagent 数量和证据保留范围 |
| 输出过长 | 用户只需要决策或下一步 | 先给结论和行动，细节按需展开 |

## 核心压缩策略 (The Compression Playbook)
1. **AST / 代码骨架提取**：
   - 绝不裸读大文件！必须优先使用 `grep` / `rg` 查找类定义 (`class`)、函数签名 (`def`, `function`) 和导出模块。
   - 保留 Docstrings 和 Imports，剥离方法实现体，以获得极高压缩比的架构骨架。
2. **大 JSON 压扁 (Smart Crushing)**：
   - 在将获取的冗长 JSON 灌入 LLM 思考区之前，必须使用 `jq` 或临时 python 脚本，过滤掉全部 `null` 值、空数组和深层无用键，只保留关键叶子节点。
3. **极简闭嘴法 (Verbosity Steering)**：
   - 这是**最昂贵的一环（输出 Token）**。在压缩模式下，强制遵循不寒暄原则。
   - 绝不说“好的，让我来分析...”、“如下所示”。
   - 如果当前回合只是读取了一个文件或执行了一次测验，只回复“已阅”或“OK”，绝不总结你刚刚看了什么！
4. **可逆溯源 (CCR)**：
   - 如果你向上下文里注入了压缩后的信息，必须保留可追溯的文件标识；公开输出时优先使用仓库相对路径，避免泄露家目录、账号名或机器路径。如果后续发现需要具体某一行实现，再去切片化抓取。

## 循环和动态工作流预算

Loop Engineering 或动态工作流启动前，必须先回答“这件事值得循环吗”。如果只是一次性小任务，直接用普通计划完成，不进入 subagent 编排。

预算字段：

1. **max rounds**：最多循环几轮，达到上限仍未通过就停止并报告 blocker。
2. **time cap**：最长运行时间，超时必须汇总已完成证据，不继续拖延。
3. **Token cap**：每轮只保留任务目标、关键证据、当前 diff、验证结果和未决风险；子任务输出先摘要再进入主上下文。
4. **subagent cap**：限制并行子 Agent 数量；简单任务不得为了形式感拆成多 Agent。
5. **tool scope**：列出允许工具和禁止工具；安装、联网、写入、外部执行仍需单独授权。
6. **checkpoint**：每轮结束有可读检查点，包含继续、停止、回滚或升级为人工判断的条件。

压缩规则：

- 子 Agent 返回内容不得原样全贴进主上下文；只保留结论、证据路径、失败原因、下一步。
- 主循环只保留当前轮需要的信息，旧轮细节沉淀到 SOP、memory 或报告附件。
- 声称节省成本时，必须标明是实测还是预估；没有测量就不写确定节省比例。

## 工具与生态观察清单

这些工具只作为候选处方，不自动安装、不自动信任：

- **Headroom**：上下文压缩中间件。业务含义：在文本进入模型前先瘦身。使用前必须由 `ecosystem-scout` 和 `security-gate` 做 intake。
- **Serena MCP**：语义代码检索 MCP。业务含义：按符号和语义找相关代码，减少整仓读取。适合大仓库摸底和重构前定位。
- **Aider repo map**：仓库地图策略。业务含义：按 Token 预算挑关键文件、类和函数签名，不直接把仓库全塞进上下文。
- **Repomix / Gitingest**：仓库打包和 Token 统计。业务含义：把仓库变成可给模型读取的包，但必须配 ignore、安全扫描和大小预算。
- **Context7**：最新文档检索。业务含义：只取当前库、版本和 API 相关材料，降低过时 API 幻觉。
- **MCP compressor / Context Mode**：压缩工具定义或用“搜索 + 执行”替代海量工具 schema。业务含义：少暴露工具面，把 Token 留给任务证据。

## 第三方工具快速接入模式

当内置的 `rg`、`jq`、切片读取、代码骨架和手工压缩已经不够，且第三方工具能明显降低成本、时间或错误率时，可以推荐并协助快速安装和应用。但这不是自动安装权。

只有同时满足这些条件，才推荐第三方工具：

1. **收益明确**：能明显减少 Token、缩短定位时间、降低漏读风险，或提供当前工具没有的真实产物。
2. **任务稳定**：输入类型稳定，例如大仓库语义检索、重复仓库打包、MCP 工具面过大、长文档定点检索。
3. **可验证**：安装后能用一个小样本证明它确实能产出结果，并能记录节省口径。
4. **可回退**：不修改项目核心文件，或能清楚说明改了什么、怎么卸载、怎么不用它继续完成任务。
5. **来源可信**：优先官方仓库、活跃维护项目或已经登记在 `skills_manifest.json` 的可选依赖。

快速接入流程：

1. **先推荐**：给出 1-3 个候选，说明解决什么问题、业务/用户价值、主要工作、成本风险限制、需要谁配合。
2. **先过门**：转 `ecosystem-scout` 查来源和替代品；转 `security-gate` 做本地启发式预检。
3. **先展示命令**：安装前必须展示命令、工作目录、会写入的位置、是否联网、如何回退。
4. **等明确授权**：只有用户用祈使句授权，例如“安装吧”“用 Serena 接一下”，才执行安装或配置。确认式提问不算授权。
5. **前台执行**：安装、克隆、初始化、索引和 smoke test 都在前台完成，不开长后台任务。
6. **小样本试用**：先跑最小样本，证明工具能解决当前问题，再进入正式任务。
7. **留下口径**：记录安装方式、版本/来源、适用场景、节省证据、风险和不再使用时的退出方法。

## 协作边界

- 外部工具、MCP、第三方 Skill intake -> 先转 `ecosystem-scout`，再过 `security-gate`。
- 最新官方资料、API、库版本 -> 转 `research-and-sourcing`，但要求它定点检索，不整篇搬运。
- 真实成本、Token、Grounding 或计费 Query 统计 -> 转 `ai-cost-grounding-measurement`。
- 长期经验沉淀 -> 转 `memory-bank`，保留症状、根因、解法和防范，不贴大段日志。

## 验证门

只有通过以下检查，才可以声称“节省了 Token”：

1. **量化口径**：写清原始输入量、压缩后输入量、估算方法；没有 tokenizer 实测就标注“预估”。
2. **证据保真**：压缩后仍保留文件、行号、字段、URL、命令或来源线索。
3. **质量回看**：回答没有因为压缩漏掉关键错误、权限边界、安全风险或用户约束。
4. **安全过滤**：不得把 API key、token、cookie、私人路径、真实账号和无关日志带入输出。
5. **不新增摩擦**：不得为了省 Token 安装重型依赖；外部工具只作为可选处方。

## 反模式

- 为了省 Token 删除风险、证据缺口或授权边界。
- 把“80% 节省”这类数字写成承诺，但没有本地任务实测。
- 直接让 Agent 读取整仓、整页文档、完整 MCP schema 或完整日志后再总结。
- 把压缩后的摘要当成事实源，后续需要精确判断时不回查原始材料。
- 用外部打包工具处理私有仓库前，不配置 ignore、不做安全扫描、不说明数据边界。
- 把“推荐工具”当成“已经获准安装”；没有明确授权时，只能给安装方案和使用步骤。

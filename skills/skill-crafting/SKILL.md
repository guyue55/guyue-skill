# skill-crafting（技能制作与封装决策）

高质量 Skill 不是更长的提示词，而是一份可发现、可激活、可执行、可验证、可卸载的行为契约。先证明工作流值得封装，再选择最窄形态；现有能力能修好时，不制造影子技能。

## 首次 Trace

首次接管时输出一次：

`[Trace: Guyue/SkillCrafting] 验证复用价值、封装形态、路由边界与评测证据`

只有候选形态、授权边界或验证结论实质变化时追加，不逐条暴露内部推演。

## 一、先判断是否值得封装

只有满足以下条件才继续：

1. 工作至少发生两次；或虽次数少，但明显会再次发生且单次成本、错误代价或交接成本很高。
2. 输入可以稳定描述，而不是每次都从完全不同的问题重新探索。
3. 核心步骤可重复，且知道哪些步骤会随项目变化。
4. 输出有可观察的通过标准、失败状态和停止条件。
5. 维护收益大于路由、文档、测试和版本治理成本。

证据可以来自用户提供的会话、任务记录、代码、发布事故、SOP、官方规范或已跑通的操作。真实会话是高价值来源，但不是制作所有 Skill 的强制前置；清晰规范和可复现工作流同样可以支撑。证据不足时先做普通计划、实验记录或 SOP，不把愿望包装成成熟能力。

## 二、选择最合适的封装形态

| 形态 | 适用条件 | 不适用 |
|---|---|---|
| Skill | 可复用工作流、操作指南、判断框架，需要被运行时按意图发现 | 只发生一次或没有稳定步骤 |
| Custom subagent | 边界明确的专家角色、可委派的只读调研或独立审查，需要隔离上下文 | 只是几条顺序固定的命令 |
| SOP | 已成功闭环、主要由人或普通 Agent 按步骤执行 | 需要自动触发或强路由 |
| Hook / Automation | 有明确事件或时间触发、幂等性、预算和外部写入边界 | 仍需大量方向判断 |
| CI gate | 规则可机器判定，目标是阻止回归 | 依赖主观体验或开放调研 |
| 普通代码/脚本 | 输入输出结构稳定，核心价值是确定性转换 | 主要价值是语义判断和协作纪律 |

组合优于万能形态：例如 Skill 负责判断和路由，脚本负责确定性检查，Custom subagent 负责独立审查，CI 负责阻止回归。

## 三、证据采集与隐私边界

1. 先读用户已提供的记录、当前仓库和现有 Skill，确认是否已有可升级入口。
2. 只有历史会改变设计时才检索会话；用户给出具体路径或明确授权扫描范围后再读取本地日志。
3. Codex JSONL 优先使用：
   `python3 scripts/codex_extractor.py <jsonl-or-dir> --cwd <project> --roles user,final --dedupe --stats`
4. Claude/Cursor 使用相应专用提取器。输出必须限长、按角色过滤、脱敏，不把系统/开发者指令、工具原始输出或整库日志塞进上下文。
5. 从记录中提取稳定输入、成功步骤、失败分支、用户反复纠正点、停止条件和可验证产物；不要只统计关键词或复制漂亮结论。

`ai_log_scanner.py` 只用于用户明确要求发现本机日志位置时的只读盘点，不是每次技能制作的固定第一步。

## 四、Skill 合同

制作 Skill 时至少定义：

- **核心工作**：一句话说明唯一职责。
- **正向触发**：用户动词、对象和上下文标记。
- **负向触发**：相邻任务、项目专用误触发和明确不处理的场景。
- **稳定输入与输出**：文件、参数、证据和产物。
- **执行步骤**：按依赖排序，区分判断步骤和确定性脚本。
- **失败与停止**：缺证据、预算耗尽、外部状态冲突和授权缺口。
- **副作用**：本地写入、网络、安装、付费、消息、发布、删除及回滚。
- **验证**：结构检查、定向测试、活体回放、独立审查和人工体验。
- **生命周期**：`discovered -> selected -> activated -> resource_read -> verified/failed`。

Frontmatter description 要短而有区分度，因为运行时通常在启动阶段加载所有名称和描述；完整流程放正文，重型参考放 `references/`，确定性逻辑放 `scripts/`。不得用 description 复述整份 Skill。

## 五、制作与升级流程

1. **复用扫描**：查当前 Skill、manifest、插件和脚本；能通过改 description、负向路由、正文或测试解决，就升级旧入口。
2. **冻结基线**：记录当前触发、输出、Token/时间、成功率和已知失败。升级任务保留至少一个旧失败样本。
3. **写合同**：先写核心工作、正负路由、输入输出、停止条件、副作用和验证，再补说明文字。
4. **按需蒸馏**：人物心智或长材料可调用 `huashu-nuwa` / `book-distiller`；Skill 可安装性和传播性可用 `luban` 审查。工具不可用时按同一证据合同继续，不伪装已调用。
5. **最小实现**：正文保持单一职责；复用仓库脚本和标准，避免复制外部 README 或创建大而全的 Skill。
6. **安全与依赖**：外部来源先过 `ecosystem-scout` 与 `security-gate`；依赖记录官方来源、许可、精确提交、安装动作和退出路径。未经具体授权不下载、安装、执行或公开写入。
7. **评测**：用结构 prompt 和机器可读行为契约覆盖正向、负向、高风险和恢复场景。
8. **活体回放**：在真实运行时比较“无 Skill 基线”和“启用 Skill”，保留未参与编写的留出样本；重要改动重复运行，记录成功率、偏差、Token/时间和副作用。
9. **安装验真**：检查完整 payload、运行时发现、真正激活和资源按需读取。文件存在、安装命令成功或根 Skill 可见都不能单独证明子技能可用。
10. **发布候选**：绑定当前提交的测试、归档、安装、活体回放和安全证据；README、版本、变更日志与真实能力同步。提交、推送、标签、发布和 marketplace 写入分别遵守具体授权。

## 六、评测合同

至少包含以下样本：

- 应触发：典型请求和边界变体。
- 不应触发：相邻技能、泛词、其他项目同名概念。
- 高风险：安装、外部写入、凭证、删除、发布、付费。
- 恢复：中断、旧证据、重复运行和副作用重放。
- 反事实基线：没有该 Skill 时会发生什么错误或成本。
- 留出样本：编写时未用于调参的真实或等价任务。

机器可读合同应分开记录 `expected_routes`、`forbidden_routes`、`required_actions`、`forbidden_side_effects` 和最低证据等级。真实回放观察必须指向证据文件并绑定哈希；结构校验只能证明合同存在，不能冒充行为通过。

## 七、Loop Engineering / Dynamic Workflow Packaging

可能驱动多阶段执行的 Skill 必须兼容 Long Goal v3：稳定 ID、承诺覆盖、停止原因与完成判定分离、副作用重放、版本化审批、委派所有权/BASE/报告/独立审查/收束预算，以及绑定文件哈希、实现版本和工作树状态的终局证据。

动态工作流必须写最大轮数、最大时间、最大 Token、最大子任务数量、工具、成本、人工检查点和回滚。先小样本回放，再决定沉淀为 Skill、Custom subagent、Hook、Automation 或 CI gate。执行器与最终检查器尽量分离；没有独立检查器和熔断条件时，不发布“自动闭环”声明。

## 八、反模式

- 不因用户说“做个 Skill”就跳过复用价值和形态判断。
- 不把所有任务都强制先扫描本机历史，也不自动读取未授权的会话目录。
- 不要求用户对已经明确委托的仓库内可逆制作重复授权；外部安装、公开写入和高风险动作仍需具体授权。
- 不用评分表、文件数量、CI 绿灯或自评替代真实回放。
- 不生成并不存在的审计脚本名、工具结果或安装成功声明。
- 不把第三方源码全文 vendoring 到本仓库，只保留必要适配、锁定指针和验证证据。
- 不把项目专用路径和泛触发词写进通用 Skill。
- 不在脏工作区混入用户改动，不把私有路径、凭证、原始会话或本地索引带进发布包。

# Guyue 数据与发布架构评审

> 状态：架构基线；第一阶段已实现
>
> 评审日期：2026-07-13
>
> 代码基线：`v1.4.0` / `7b19578eb1ade7298fd4d8bfa3a8584c2a074844`
>
> 范围：长期知识、运行状态、缓存、项目控制、公开源码与安装载荷

## 结论

原讨论的方向成立，但尚不能直接作为实施合同。最重要的判断应收敛为：

1. **按所有权和生命周期分层，不按文件格式分层。** Skill 仓库、目标项目仓库和用户目录分别拥有产品资产、项目共享资产和私人运行资产。
2. **默认使用 `~/.guyue`，支持 `GUYUE_HOME` 覆盖。** 它是 Guyue 的用户数据根目录，不是安装目录的附属文件夹。
3. **长期知识采用 Wiki-first。** Markdown + YAML Frontmatter 是可读事实源；JSON 是小规模状态或可重建索引；当前不引入 SQLite。
4. **`.guyue_memory` 进入兼容迁移期。** 新版本不再扩展该目录，新写入迁往用户目录，旧数据只读发现、显式迁移、不双写、不静默删除。
5. **公开源码、安装载荷、发布证据必须分开治理。** 建立一个发布载荷清单作为安装检查和归档校验的共同输入，不能继续让多个脚本各自维护不一致的路径列表。

这不是一次目录改名。它解决的是安装升级后私人数据丢失、多份安装形成记忆孤岛、公共与私有资产混放、缓存被误当知识、安装检查漏文件，以及长期任务状态归属不清等问题。

## 当前事实

以下结论已由 `v1.4.0` 仓库内容确认：

| 事实 | 当前证据 | 影响 |
|---|---|---|
| 私有记忆默认写入安装根目录 | `src/memory_store.py` 的 `runtime_memory_dir()` | 安装删除、插件缓存更新或多份 clone 会造成丢失或孤岛 |
| 公共索引与私有数据共用 `.guyue_memory` 名称 | `.guyue_memory/index.json`、`.gitignore`、`skills/memory-bank/SKILL.md` | Git 规则依赖“已跟踪文件不受 ignore 影响”的隐含知识 |
| 记忆读取只返回索引条目 | `src/mcp_server.py` 的 `guyue_read_memory()` | 与 Skill 所述“命中后读取 Markdown 详情”不一致 |
| GC 的 dry-run 会创建目录 | `scripts/memory_gc.py` 在判断 `dry_run` 前创建 `active/`、`archive/` | dry-run 不是零副作用 |
| `review_after` 到期会直接归档 | `scripts/memory_gc.py` 的 `archive_reason()` | “需要复查”被错误等同于“已失效” |
| 技能发现索引写入记忆目录 | `scripts/discover_local_skills.py` | 可重建缓存与不可轻易丢失的知识混为一类 |
| 完整安装检查未要求 MCP 与记忆核心文件 | `scripts/check_full_install.py` 的 `BASE_REQUIRED_PATHS` | 可能出现安装检查通过但记忆能力缺失 |
| 发布文件要求存在多份手写清单 | `scripts/check_full_install.py`、`scripts/check_birth_certificate.py`、`.gitattributes` | 清单容易漂移，缺少共同事实源 |

原讨论中“当前有大量 staged 修改”的判断已经过时。当前 `dev`、`main`、远程分支和 `v1.4.0` 指向同一提交，工作树干净，可以从该提交建立独立实施分支。

## 需要修正的讨论结论

### 1. 数据库不是当前成熟度的前提

SQLite 能解决事务和复杂查询，但会立刻引入 schema migration、备份一致性、损坏恢复和不可直接审阅等长期责任。当前更重要的是让知识可读、可迁移、可审查，并先证明真实规模和并发压力。

本阶段冻结为：

- Markdown + YAML Frontmatter：长期知识事实源。
- JSON：工作区映射、迁移收据等小型结构化状态。
- JSONL：只追加审计事件。
- 搜索索引：可删除重建，不得成为第二事实源。
- SQLite/FTS：达到明确阈值后再评估，优先作为派生索引而非权威知识库。

### 2. Wiki 只承载知识，不承载所有数据

Wiki 适合决定、经验、根因、原则和项目背景；不适合锁、缓存、原始会话、迁移中间文件、高频日志和密钥。把所有内容都做成 Markdown，同样会制造新的混乱。

### 3. `~/.guyue` 是产品选择，不冒充跨平台标准

默认统一目录有利于用户理解、备份和跨运行时共享。`GUYUE_HOME` 负责企业策略、便携安装和 XDG 等偏好。实现必须使用用户主目录 API 解析路径，不能依赖 shell 展开字面量 `~`。

### 4. 同盘快照不等于备份

`~/.guyue/snapshots/` 只能用于迁移前回滚和短期恢复。真正备份必须由导出命令写入用户指定的外部位置，并生成清单与哈希；不能因为目录名叫 `backups` 就宣称具备灾难恢复能力。

## 所有权模型

| 权威根 | 拥有内容 | 不应拥有 |
|---|---|---|
| Guyue Skill 仓库 | 公共 Skill、默认策略、schema、模板、公开知识、稳定评测合同、运行代码 | 私人知识、机器路径、原始会话、目标项目的 Goal 实例 |
| 目标项目仓库 | 团队共享的项目策略、公共 Goal 控制包、项目 ADR、可公开证据 | 当前用户私密经验、机器状态、缓存、凭证 |
| `~/.guyue` | 用户差异配置、私人知识、工作区状态、缓存、迁移收据、受控快照 | Skill 公共知识副本、项目公共事实副本、密钥明文 |

资产归属先看“谁维护、是否共享、是否应随版本演进”，再看“由谁生成”。运行时生成的公共发布收据仍可进入仓库；人工编写的私人经验仍不应进入仓库。

### 项目配置

项目公共配置统一使用根目录下可选的 `.guyue.toml`：

- 可以进入 Git，但不得包含秘密或机器绝对路径。
- 只保存项目身份、允许的知识作用域、保留策略等团队共享差异。
- 不复制 Guyue 的完整默认配置。
- 不再同时引入 `.guyue/project.toml`，避免两个项目配置入口。

默认不在目标项目中创建 `.guyue/workspace.json`。工作区映射由 `~/.guyue/state/` 管理，避免 Agent 未经要求污染每个目标仓库。

## 身份与作用域

### `project-id`

- 只在项目显式提供 `.guyue.toml` 时成立。
- 用于跨 clone 共享同一项目的私人知识。
- 必须是稳定、非敏感、可移植的标识，不从本机绝对路径直接生成公开 ID。

### `workspace-id`

- 标识当前用户机器上的一个 checkout。
- 首次使用时随机生成，映射保存在用户状态目录，不进入 Git。
- 没有显式 `project-id` 时，只启用 workspace 作用域，不猜测两个 clone 是否属于同一项目。

### 知识作用域

| 作用域 | 来源 | 可写方 | 是否发布 |
|---|---|---|---|
| `public` | 当前激活的 Skill 包或目标项目仓库 | 发布/项目维护流程 | 是 |
| `global` | `~/.guyue/knowledge/global/` | 当前用户 | 否 |
| `project` | `~/.guyue/knowledge/projects/<project-id>/` | 当前用户 | 否 |
| `workspace` | 与工作区状态绑定的私人经验 | 当前用户 | 否 |
| `inbox` | 尚未验证的候选知识 | Agent 或用户 | 否 |

公共知识不复制到 HOME。读取时从当前激活版本直接加载，从而让知识版本与 Skill 版本自然绑定。

检索结果是带来源和置信度的并集，不是简单的“最近作用域覆盖最远作用域”。发生冲突时，返回冲突和证据新鲜度；安全底线始终采用最严格规则。

## 知识写入合同

每个知识页面至少包含：

- `schema_version`、稳定 `id`、`kind`、`status`；
- `scope`、可选 `project_id` / `workspace_id`；
- `revision`、`content_hash`；
- `created_at`、`updated_at`、`last_verified_at`、`review_after`；
- `confidence`、`sensitivity`、`tags`；
- `supersedes`、`evidence_refs`；
- 正文中的现象、根因、处理、预防或决定理由。

状态机固定为：

```text
inbox -> active -> needs_review -> active
                     |             |
                     +-> superseded+-> archived
```

- `review_after` 到期只进入 `needs_review`，不得自动归档。
- `superseded` 必须指向替代条目并保持无环。
- `archived` 保留完整内容和原因，不以截断正文代替归档。
- 未验证信息只能进入 `inbox`，不能伪装为高置信长期知识。

### 并发写入

第一阶段不承诺任意网络文件系统上的并发写入，只支持本机文件系统。每次更新必须：

1. 读取目标页并记录 `revision` 与 `content_hash`。
2. 获取每个目标页的排他锁；锁必须有拥有者、创建时间和超时恢复规则。
3. 写入同目录临时文件，刷新内容后再次核对旧哈希。
4. 哈希不一致则返回冲突，不做 last-write-wins。
5. 原子替换后递增 revision，再释放锁。

不同 ID 的新增页可以并行；同一页正文冲突必须重新基线或人工审查。

## 配置与安全合同

配置优先级：

```text
命令行临时参数 > 项目差异配置 > 用户差异配置 > Skill 默认配置
```

安全规则不服从普通覆盖优先级，而是所有层中最严格的约束生效。

- `config.toml` 只保存差异和凭证引用名，不保存 Token、cookie、私钥或密码。
- 新建用户根目录和私人知识目录时，应尽可能设置为仅当前用户可访问。
- `GUYUE_HOME` 必须规范化并检查可写性、符号链接和路径逃逸。
- 遇到未知的未来 schema 版本时 fail-safe：只读或拒绝，不猜测迁移。
- 原始聊天、完整终端日志和浏览器记录默认不进入长期知识。

## Git 与发布边界

| 资产 | 公开 Git | 安装载荷 | Release 证据 |
|---|---:|---:|---:|
| Skill、运行代码、schema、模板 | 是 | 是 | 可选 |
| 当前版本必要的公开知识 | 是 | 是 | 可选 |
| 稳定评测合同与必要验证器 | 是 | 视运行需要 | 可选 |
| 原始活体运行记录 | 脱敏精选后可选 | 否 | 是 |
| 历史审计与维护计划 | 可选 | 否 | 可选 |
| 私人知识、状态、缓存、密钥 | 否 | 否 | 否 |

`release-manifest.json` 是运行载荷的声明式事实源，名称不再与 `package-manifest.json` 混用；`release-payload.lock.json` 是由当前候选生成的精确文件与哈希锁。前者稳定表达政策，后者绑定版本产物。

- schema 与产品版本；
- 必需文件和允许的目录规则；
- runtime 特有文件；
- 禁止进入载荷的路径类别；
- 展开后的精确文件清单与哈希生成规则。

`scripts/check_full_install.py`、`scripts/check_birth_certificate.py` 和归档检查共同消费该清单；`.gitattributes export-ignore` 只作为第二道防线。发布收据记录锁定文件数、聚合内容哈希、来源 commit 和 runtime，不把可移动分支名当作可复现身份。锁文件在最终源码变更后由 `python3 scripts/build_release_lock.py` 重新生成。

## 兼容迁移合同

建议从 `v1.4.0` 之后的两个次版本提供旧 `.guyue_memory` 只读兼容。精确结束版本必须在实际实施版本确定后写入文档，不能只写“未来两个版本”。

迁移遵循：

1. **发现**：扫描旧公共索引、旧私有索引、详情文件和技能发现缓存，只生成报告。
2. **计划**：分类为公共知识、私人知识、缓存、重复项、损坏项和敏感项。
3. **备份**：在 `snapshots/` 生成迁移前快照、清单与哈希。
4. **执行**：显式命令写入新路径；默认不自动迁移、不双写。
5. **对账**：逐项核对数量、ID、正文哈希、状态和证据引用。
6. **切换**：新路径单写，旧路径只读。
7. **回滚**：失败时删除本轮新写入并从快照恢复；旧数据始终保留。
8. **退役**：兼容期结束前给出 Doctor 警告；删除旧数据必须再次获得用户明确授权。

重复项按稳定 ID 和正文哈希判断。相同 ID、不同正文必须标为冲突，不得自动选择较新的文件时间。

## 分阶段执行计划

### P0：冻结合同

- 将本文转为正式 ADR，并记录最终批准状态。
- 确定实际实施版本和旧路径兼容结束版本。
- 固定 `release-manifest.json` schema 与职责。
- 为路径、身份、知识页面和迁移收据定义最小 schema。

退出条件：所有路径、所有者、写入方、读取方、迁移和回滚均无双重事实源。

### P1：先修正确性

- 让 MCP 命中后安全读取真实详情。
- 修复 GC dry-run 副作用。
- 将到期复查改为 `needs_review`，不直接归档。
- 增加详情逃逸、坏索引、缺文件和生命周期回归测试。

退出条件：旧架构在迁移前先做到“声明与行为一致”。

### P2：路径解耦

- 新增唯一的路径解析模块，禁止 MCP、GC、发现脚本自行拼路径。
- 新写入迁往 `GUYUE_HOME`；技能发现索引迁往 cache。
- 增加只读安装目录、空 HOME、自定义 HOME、符号链接和权限测试。
- 保留旧路径只读发现。

退出条件：安装目录只读时，读取公共知识、写入私人知识和重建缓存均能工作。

### P3：显式迁移

- 提供 `plan`、`migrate`、`verify`、`rollback`、`doctor` 五个动作。
- 验证重复执行幂等、异常中断可恢复、不同正文冲突不覆盖。
- 记录脱敏迁移收据，不记录私人正文。

退出条件：模拟旧目录中的正常、重复、损坏、敏感和冲突数据，迁移均不丢失、不泄露、不静默覆盖。

### P4：统一发布载荷

- 引入 `release-manifest.json`。
- 让安装检查、出生证和归档验证共同消费它。
- 补齐 MCP、memory、GC、公共知识和迁移工具的载荷要求。
- 在空 HOME、无 `.git`、只读安装目录中验证精确候选。

退出条件：源码、Git archive 和安装目录对同一 runtime 产生一致的载荷文件集合与哈希。

### P5：稳定后再降冲突

- 评估拆分集中式评测合同、测试 Prompt 和公共知识索引。
- 评估 Skill 自有 route/eval/reference 文件与聚合生成器。
- 只有真实冲突频率证明收益时，才引入 Changelog fragments 或 SQLite 搜索缓存。

## 验收矩阵

| 场景 | 必须证明 |
|---|---|
| 只读安装目录 | 不向安装目录写入，私人知识仍可保存 |
| 多份 Guyue 安装 | 共享同一用户数据根，但公共知识跟随各自激活版本 |
| 两个 clone、无项目配置 | workspace 隔离，不擅自合并项目知识 |
| 两个 clone、相同显式 `project-id` | 可共享项目私人知识，工作区状态仍隔离 |
| 两进程写同一页 | 一个成功，另一个得到可诊断冲突，不丢更新 |
| `--dry-run` | 文件系统零变化 |
| 复查日期到期 | 进入 `needs_review`，正文仍可读 |
| 删除 cache | 启动正常且索引可重建 |
| 旧数据迁移中断 | 旧数据不变，新数据可回滚，重跑幂等 |
| 未知 schema | 只读或拒绝并给出升级提示，不猜测解析 |
| 发布候选 | 精确载荷含全部运行依赖且不含私人状态 |
| 安全扫描 | 无密钥、私人绝对路径、原始聊天或机器索引泄露 |

## 明确不做

本轮架构升级不包含：

- SQLite 权威存储或向量数据库；
- 多设备同步和云端记忆服务；

## 最终建议

可以放行分阶段实施，但不应把原讨论原样当作需求文档。本文补齐后的基线先解决“事实与设想混写、目录命名摇摆、身份模糊、并发协议缺失、兼容期限不可执行、发布清单多头维护”六个问题。

真正的第一步不是搬目录，而是让旧实现的读取、dry-run 和生命周期先符合已经公开的能力声明；随后再做路径解耦与显式迁移。这样每一阶段都有独立价值，也能在失败时回退到 `v1.4.0`，不会把一次必要升级变成不可审计的大爆炸重构。

# 古月首轮体验打磨报告（2026-07-11）

## 1. 验料结果（Skill 前提挑战）

- 挑战 1，真实问题：成立。复杂 Agent 工作常见的失败不是不会生成内容，而是需求未闭合、边界漂移、状态灯代替活体、阶段完成冒充目标完成。
- 挑战 2，独特角度：来自真实协作记录、26 个窄能力路由、长任务控制包和可复跑门禁，不来自人格口号或 Skill 数量。
- 挑战 3，安装理由：临时提示无法稳定提供包体收据、项目误触门、证据哈希、恢复协议和失败注入；完整仓库可以。
- 挑战 4，公共传播性：原有能力可展示，但第一份真实结果出现太晚；本轮把传播钩子收束为“先定边界、再执行、用证据收尾”，并增加 30 秒只读验货。

验料结论：好料，继续精雕；不扩套件。

## 3. 生态位判断

- 纵向结论：古月已从人格提示词演进为 Personal Agent Operating Layer；下一步应缩短“理解价值到看到证据”的距离。
- 横向结论：同行的首屏优势主要来自窄入口、一条命令、可见产物和诚实边界，而不是功能目录更长。
- 交叉洞察：古月真正该抢的不是“万能 Agent”生态位，而是中文、证据优先、可恢复、Anti-Bloat 的个人 Agent 操作层。
- 一句话新定位：让 Agent 在复杂任务里先定边界、再执行、用证据收尾。

## 5. 差距清单

### P0：不补就不能公开或信任

- 当前没有阻断本地打磨交付的 P0。仍禁止把 1 条成功回放写成 17 条模型行为全部通过，也禁止把未发布工作树写成公开版本。

### P1：补上后明显提升安装与传播

- 通用 Skills CLI 的根级复制仍不能保证完整载荷，需继续使用完整仓库挂载和安装收据。
- 公开 `main` 在下一次获授权发布前看不到本轮首屏改进；本报告不把本地工作树写成已发布状态。
- 其余 16 条合同与其他运行时仍缺本轮版本的哈希绑定活体覆盖。

### P2：下一轮再做

- 在不同运行时补相同留出 prompt 的发现、激活和行为对账。
- 有真实用户反馈后再决定是否缩短中段能力目录，不凭审美删除检索入口。

与同行相比最缺的三件事：一条通用完整安装命令、多运行时留出回放覆盖、真实外部用户反馈。

最有机会打穿的三件事：证据边界同屏展示、项目专属能力不误触、长任务完成证据可复跑。

## 6. 三个打磨方向

### 方案 A：细修

只改首屏文案。成本低，但旧 Showcase 和包体依赖缺口仍会制造假信任。

### 方案 B：精雕

新增一个只读确定性验货入口，复用现有能力；同步 README、GIF、VHS、示例、安装、评测和发布门。收益可见、边界清楚、没有新框架依赖。

### 方案 C：开套件

新增独立 Demo 应用或多 Agent 演示环境。展示更丰富，但安装、维护和真实性成本超过当前收益。

推荐并已执行方案 B。用户要求继续打磨，现有基础较好，方案 B 是最小但足够有力的产品改进；公开发布仍需单独授权。

## 8. README 与 Showcase 升级

- README 首屏现在按“定位 -> 安全边界 -> GIF -> 30 秒命令 -> 真实输出 -> 能力解释”展开。
- 示例输出明确区分 `deterministic_local_proof`、`runtime_activation_verified` 和 `model_behavior_verified`。
- Showcase 展示普通权限需求与带 NexusFlow 项目标记的差异，证明项目能力不会因泛关键词误触。
- VHS tape 直接运行验货脚本；内置 GIF 渲染器让没有 VHS 的环境也能复现同一叙事。
- 演示校验不再只看文件存在或 GIF 头，而会拒绝尺寸错误、缺帧、稀疏画面和重复帧。

# Long Goal Protocol

长程 Goal 协议用于处理跨多阶段、可能经历上下文压缩、恢复、中断和多轮验证的任务。它来自真实项目推进经验：短 Goal 指向长文档，长文档驱动阶段执行，执行账本负责恢复，活体证据负责验收。

协议分成两个不能混用的状态：

- **铸造阶段**：把模糊愿景变成经过用户确认、项目证据支持、可恢复和可验收的控制包。只做调查、澄清、方案、规划和治理准备，不实现目标功能。
- **执行阶段**：只接受已经通过决策关闭门和独立就绪审查的控制包，从账本记录的下一入口持续推进。

## 适用场景

- 用户要求“制定计划并一一执行，直到全部完成”。
- 任务包含多个里程碑、阶段或批次。
- 完成标准依赖真实构建、浏览器验收、截图、报告、导出包、人工体验量表或发布候选证据。
- 上下文可能超过一次会话，需要恢复时不跑偏。

## 铸造阶段

### 先查项目现场

在向用户提问前，按项目可用情况读取：

1. 仓库指令、README、架构文档、需求文档和现有计划。
2. 当前分支、工作树、近期提交、相关会话摘要和可公开记忆。
3. 运行入口、数据与权限事实源、核心模块、测试与发布脚本。
4. 当前构建、报告、截图、部署或发布候选证据，并判断是否新鲜。
5. 已有同类功能、可复用标准件、历史失败和明确非目标。

把调查结论先分成已确认、推断、冲突和待决策四类。控制包就绪前再落成四种可执行认知：`VERIFIED_FACT`、`FROZEN_DECISION`、`HYPOTHESIS`、`EXPERIMENT`。推断不能伪装成事实或用户承诺，假设必须绑定可证伪实验、通过标准和失败后的删除或改写路径。

调查采用分层预算。决策未关闭的单轮只允许最多 4 次定向读取或检索，加 1 次轻量状态探针；单文件最多读取 120 行，工具返回合计不超过 16000 字符。优先确认项目身份、当前状态和已有目标路径，不预读控制包模板、完整原则、完整 manifest、测试、发布档案或全仓正文；不得运行完整测试套件、安全扫描、安装、构建或 live replay。只有当某项重操作直接决定当前最高影响问题或用于处置安全风险时才可例外，并记录必要性。证据已经足以形成下一问时立即停止工具调用，不为“显得充分”继续耗费上下文。

### 逐项确认

只追问会改变目标、范围、方案、预算、授权或完成定义的问题。每轮只问一个最高影响问题，并写清：

- 当前证据或冲突；
- 不确认会影响什么；
- 推荐默认值及理由；
- 2-3 个清晰可选方向。

用户回答后更新决策记录，再决定下一问。用户说“赶时间”“不要问”“直接开始”不构成决策证据；“行业最好”“全面完善”等最高级表达也不能直接转成验收标准。

铸造不是可再次委派的执行目标。当前会话必须完成调查和逐项确认；决策未关闭时，只能直接向用户提出一个问题，不能输出“启动 Forge”“生成总控方案”或让下一任务继续准备的一行提示词。

### 决策关闭门

只有以下内容全部明确，才允许生成控制包：

- 真实用户、核心场景、业务价值、当前替代方式和不做的后果；
- 当前基线、目标状态、量化或可观察结果、范围和非目标；
- 时间、成本、工具、数据、权限、合规、外部服务和不可逆动作边界；
- 对比方案、推荐方案、关键取舍、事实源、架构边界和异常兜底；
- 里程碑、依赖、阶段产物、检查器、停止条件、预算、恢复和回滚；
- 最终完成定义、否定清单、活体证据、证据新鲜度和人工检查点。
- 终极愿景、本 Goal 有限交付和必须靠时间验证的结果已经分开；控制包被新事实证伪时如何暂停、追加控制修订和撤销旧完成口径。
- 最危险的用户结果已有先于规模化建设的纵向切片和风险门；同一设计门连续失败时有明确的设计复核熔断。

仍有会改变方向的待决策项时，继续逐项确认。不得把未决问题塞进最终提示词，交给执行阶段自行猜测。

### 生成控制包

优先沿用项目现有文档目录和命名；没有约定时，在一个仓库相对目录中生成：

```text
docs/goals/<goal-slug>/
├── goal-master.md
├── execution-ledger.md
├── phases/
│   └── phase-01-<name>.md
└── evidence/
    └── index.md
```

字段和写法以 [templates/long-goal-control-pack.md](templates/long-goal-control-pack.md) 为准。新控制包使用格式版本 4：继承 v3 的稳定 ID、委派收束和哈希证据，再增加权威顺序、三层时间尺度、认知与实验台账、控制修订、纵向风险门和 A/B/C 终局封账。`控制包版本`只表示文件格式；`当前控制修订`表示某项 Goal 的方向版本。版本 2/3 只保留历史解析兼容，不具备 v4 的元控制门禁。总控文档是唯一控制面，必须逐个列出账本、每个阶段计划和证据索引，不能只引用阶段目录；活跃控制文档要有数量上限和单一职责，聊天记录不是事实源。生成后运行 `python3 <guyue-root>/scripts/check_long_goal_pack.py --repo-root <target-repo> --mode ready <goal-master.md>`，检查引用、状态机、三层时间尺度、可证伪实验、风险门、控制修订、承诺与证据双向引用、预算、委派包、审批版本、重放类别、恢复入口和未替换模板字段。`<guyue-root>` 是古月完整安装目录，`<target-repo>` 是承载控制包的真实项目根；检查器会拒绝逃出目标项目的总控路径。

### 元控制与独立判断

总控是可被证伪的工作假说，不是上帝文件。古月要同时做两次判断：向上检查当前任务是否仍服务于用户价值和本 Goal 交付，向下检查抽象原则是否已落成一个可观察的真实切片。发现新事实与总控冲突时，先保留冲突并暂停受影响范围；不得为了继续执行而静默改目标、降门禁或新增一轮控制文档。只有用户批准绑定新基线、`ACT-` 和 `DEC-` 证据的新 `REV-` 控制修订后，才能恢复；这不是把格式版本从 v4 随意改成 v5。

历史计划、旧阶段绿灯和旧“完成”结论默认没有继承权。若确需保留，必须逐项说明证据为何仍新鲜、仍覆盖当前承诺；检查器通过只能说明控制包结构完整，不能证明产品、体验或终极愿景完成。

执行前先选择最危险且最能代表用户价值的纵向切片，使一个真实输入贯穿事实源、机制、界面或输出、失败降级和证据复算。风险门未通过前，不批量扩张页面、内容、Agent、依赖或基础设施。每次失败必须在账本绑定当前 `REV-`、风险门、假设、独立实验 ID、差异说明和结果为失败的 `ATTEMPT` 证据；同一控制修订与风险门积累三个不同实验、三条不同差异说明和三个不同失败证据时，状态必须写入 `BLOCKED_DESIGN_REVIEW_REQUIRED`，停止追加实现，回到设计复核和用户判断。恢复时保留原失败记录，追加经批准的新控制修订并记录阻塞到执行中的状态转换。

### 状态、承诺与重放

- **状态机**：正向状态只能沿 `铸造中 -> 就绪待执行 -> 执行中 -> 终局候选 -> 完成` 推进；执行可进入 `阻塞`，解除阻塞后回到 `执行中`。每次变化写入账本，不靠聊天推断。
- **停止不等于完成**：`停止原因` 说明本轮为何停下，`完成判定` 说明 Goal 是否满足终局定义。预算耗尽、等待授权、工具失败和阶段结束都不能自动得到“通过”。
- **承诺覆盖**：需求、非目标、决定和风险分别用 `REQ-`、`NREQ-`、`DEC-`、`RISK-` ID，映射到阶段和 `EVID-` 证据。每个证据行只覆盖一个承诺；同一物理产物可用多个独立 `EVID-` 行复用路径与哈希。矩阵中的证据必须存在，证据索引也必须反向指回同一承诺；新鲜度必须以规范“新鲜”状态词开头，不能用“不新鲜”等子串绕过；`complete` 只能由 `FINAL` 证据放行，失败尝试用 `ATTEMPT` 留痕。发现新承诺时先补矩阵，再扩阶段；不得让交付口径悄悄膨胀。
- **副作用重放**：每个 `TASK-` 标记 `replay_safe`、`verify_before_repeat`、`compensate` 或 `human_required`。恢复时先核对外部状态、幂等键、补偿和授权，不把“继续”解释成重复执行外部写入。
- **版本化审批**：高风险动作使用 `ACT-` ID，审批绑定参数、计划版本或摘要哈希和失效条件。`APPROVED` 引用的 `DEC-` 必须由新鲜、通过的 `FINAL` 证据支持，失败 `ATTEMPT` 不能充当授权。计划实质变化、超过有效期或目标环境改变后必须重新审批。
- **认知分栏**：事实记录可复核现状，决定记录当前控制修订冻结边界，假设记录尚未证明的因果，实验只验证对应假设。实验通过不得外推成整个 Goal 或愿景完成。

### 委派与收束

只把边界独立、所有权不冲突、能由稳定输入验收的任务委派出去。每个委派项必须形成可落盘的任务包，至少包含 TASK ID、Brief、可改范围、明确不做、BASE 版本、报告路径、审查包和轮数/时间/Token 上限；返回状态限定为 `DONE`、`DONE_WITH_CONCERNS`、`NEEDS_CONTEXT` 或 `BLOCKED`。

并发不是进度。主控按固定心跳读取落盘报告和新证据；连续两次没有新证据、重复调查相同问题、越过所有权边界或耗尽预算时，必须停止扩张、取消重复任务并收束现有结论。实现审查使用 `BASE..HEAD` 或等价工作树指纹，规格审查与质量审查不得只复述实现者报告；聊天里的“做完了”不是交付物。

### 独立就绪审查

控制包完成后，切换到未参与起草的审查视角，或调用适合的独立审查能力，逐项核对：

- 每个需求、非目标、风险和用户决定是否在阶段或门禁中有落点；
- 方案是否符合当前项目，而不是只在抽象层面正确；
- 阶段是否有稳定输入、明确产物、依赖、验证命令和失败处理；
- 自动循环是否有最大轮数、时间、Token、子任务、工具和成本边界；
- 高风险、不可逆、付费、外部发送、发布、部署和合并动作是否保留人工授权点；
- 最终完成是否同时要求正向门禁、否定清单、最新活体证据和人工体验判断；
- 中断、上下文压缩、旧服务、旧报告和失败重试后能否从账本恢复。
- 总控是否明确三层时间尺度、权威顺序、历史完成权和自我推翻条件，且没有把检查器通过写成产品完成。
- 是否先用纵向切片验证最高风险用户结果，再允许规模化；三次差异化失败后是否会真正熔断进入设计复核。
- 每个用户承诺是否映射到阶段和证据，是否还有“未覆盖”项。
- 每个有副作用的任务是否定义重放、核验、补偿或重新审批方式。
- 委派是否绑定基线、所有权、报告和独立审查，且存在并发、心跳与无进展收束上限。
- 每条终局证据是否为 `FINAL`、能读取真实文件并复算 SHA-256，是否记录实现版本、工作树状态、命令与退出码；承诺与证据能否双向对账。
- A（实现）、B（证据）、C（只改总控与账本）是否为直接父子链，是否使用完整 object ID 与 `derived@master+ledger`，所有阶段是否完成，每条 `FINAL` 是否来自 `clean@A`，B 后提交历史是否从未触碰终局证据。
- 安全扫描是否覆盖控制包的新文件；新文件尚未跟踪时，不得把只扫描 `git ls-files` 的结果写成已覆盖。

审查存在阻塞项时回到逐项确认或控制包修订，不得为了交出一行提示词降低标准。

### 一行交接契约

通过独立就绪审查后，最终回复必须只有一个物理行，不使用代码围栏、标题、前言、解释或备选版本：

```text
以 <goal-master.md 的仓库相对路径> 为唯一总控，先核对控制基线、当前控制修订、三层时间尺度、认知与实验台账、风险门和执行账本，从下一入口持续推进；先证明纵向切片再扩张，若新事实证伪总控则暂停并请求批准新修订，同一设计门三次差异化实验失败则进入 BLOCKED_DESIGN_REVIEW_REQUIRED 并保留 ATTEMPT 证据；恢复时按副作用重放类别处理，终局按 A（实现）、B（FINAL 证据）、C（仅总控与账本）封账并独立复算，只有本 Goal 承诺、否定清单和人工检查点全部通过后才标记 Goal complete，且不得外推为终极愿景或时间型结果完成。
```

这行只负责启动执行，不复述全部需求，不携带未决问题，也不声称准备阶段已经完成了任何实现。

## 四件套

| 资产 | 作用 | 最低要求 |
|---|---|---|
| 总控文档 | 承载完整执行协议 | 权威顺序、三层时间尺度、认知实验、风险门、状态机、承诺覆盖、阶段顺序、委派收束、停止条件、完成定义 |
| 执行账本 | 支持恢复和审计 | 状态转换、停止原因、完成判定、失败命令、证据路径、下一入口 |
| 阶段计划 | 控制每一刀范围 | 稳定任务 ID、输入、改动边界、重放类别、定向检查、提交要求 |
| 活体证据包 | 证明当前产物真实可用 | 稳定证据 ID、覆盖承诺、文件 SHA-256、实现版本、工作树状态、命令退出码、人工判断 |

## 执行阶段

1. 读取总控文档、执行账本和当前阶段计划。
2. 确认工作树状态，分离本轮改动、用户已有改动和证据刷新噪音。
3. 只做当前阶段的最小完整切片；需要委派时先落盘任务包并锁定文件所有权与 BASE 版本。
4. 运行阶段定向检查，再运行主线门禁。
5. 拉活体证据：能打开就浏览器验收，能截图就截图，能导出就检查导出物，能回放就回放。
6. 把失败、修复、证据文件哈希、实现版本、工作树状态、残余风险和下一入口写回账本。
7. 中断恢复或重试前，按任务的副作用重放类别先核验外部状态、补偿或重新审批。
8. 需要提交时，按任务切片暂存并使用中文 conventional commit。
9. 主控按心跳检查委派报告；无新证据或预算耗尽时停止扩张并收束，不无限续派。
10. 终局先提交实现 A；所有阶段此时必须已经完成，主控处于终局候选。再从 `clean@<A 的完整 object ID>` 重建和采集 `FINAL` 证据，只把证据、报告和审查产物提交为 B。
11. B 后只把总控与账本更新为完成并提交 C，在总控写入 A、B 的完整 object ID 和 `derived@master+ledger`。运行 `python3 <guyue-root>/scripts/check_long_goal_pack.py --repo-root <target-repo> --mode complete <goal-master.md>`；只有 A -> B -> C、承诺与证据、证据哈希、阶段状态、否定清单和最终完成定义全部满足，才允许标记 complete。C 后只有从未触碰终局证据的无关提交不会破坏历史封账；先改写再还原也会被提交历史识别为失败。

## 否定清单

长程任务的完成定义必须包含“不能出现什么”。常见否定项：

- 白屏、遮挡、死链、移动端首屏关键内容不可见。
- 前端承担权限事实，或私密数据进入公开索引、公开构建产物、AI 上下文。
- 默认播放音频、默认启用重型视觉依赖、无预算引入新资产。
- 测试通过但报告、截图、构建产物或索引早于源码/数据变更。
- 阶段完成被误称为 production-ready、终局完成或真实上线。

## 证据新鲜度

旧证据是长程任务里最容易骗过 Agent 的东西。以下情况必须重跑：

- 源码、数据、配置或内容文件晚于截图、报告、索引、导出包。
- dev server 长时间运行，且刚改过路由、layout、构建配置或静态生成数据。
- production build 被清理、并行构建或并行检查污染过。
- 报告只更新时间戳，没有覆盖新路由、新场景或新断言。

## 恢复入口

上下文压缩、会话中断或用户说“继续”时，按顺序读取：

1. 总控文档的范围和最终完成定义。
2. 执行账本的当前阶段、失败记录和下一入口。
3. 当前任务的副作用重放类别、最近外部状态和审批版本。
4. 最近提交和当前工作树状态。
5. 当前阶段计划和应跑门禁。

不要从聊天残影里猜“应该做到哪了”。

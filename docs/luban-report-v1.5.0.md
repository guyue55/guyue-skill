# 鲁班审查：Guyue v1.5.0 发布候选

日期：2026-07-13
对象：古月根路由、26 个子 Skill、协作合同、数据分层、安装与发布载荷
结论：候选载荷可进入发布前封板；`main` 合并、tag 与公开发布尚未授权，也尚未完成

## 验料

这块料值得继续雕。古月真正的价值不是 Skill 数量，而是把模糊目标收敛、窄能力路由、长线恢复、证据封套和授权边界组织成一层可复跑的协作控制面。v1.5.0 的升级没有继续堆能力，而是补齐“能力之间如何协作”和“安装载荷与私人数据如何分离”两条主梁。

## 真实问题与修复

1. 原协作规则散落在根说明和少量子 Skill 中，无法证明典型任务会按正确顺序配合。现由 `collaboration_contract`、路由器和十条回归场景共同约束。
2. 完整工作流短语能命中协作合同，但旧 CLI 因没有直接 Skill 命中而提前报失败。现增加 `collaboration_candidate` 生命周期并通过活体回放。
3. 项目治理工作流可能被普通权限需求误提议。现要求 `NexusFlow`、`permissionSnapshot`、`EAC` 或 `Grounding` 上下文标记。
4. 架构文档把历史上的同线分支写成当前事实。现改为 `dev` 领先、发布前需授权快进合并的真实状态。
5. 旧数据兼容只写“未来两个版本”，不可执行。现明确覆盖 `v1.5.x` 与 `v1.6.x`，最早 `v1.7.0` 才能评估移除。

## 三个方案

- **A：继续堆新 Skill**。短期显得强，长期扩大路由碰撞、上下文和维护成本。不采用。
- **B：只写协作文档**。改动小，但无法回归，也容易在运行入口提前失败。不采用。
- **C：集中协作合同 + 路由状态机 + 机器评测 + 活体回放**。保持能力数不变，协作可解释、可验收、可演进。已采用。

## 出师边界

候选代码与文档已具备发布封板条件。真正出师仍需一条线完成：`dev` 远程 CI 绿色 -> 用户授权 -> `main` 快进 -> `main` 远程 CI 绿色 -> 空 HOME 公网安装 -> 用户授权创建 `v1.5.0` tag -> 推送 tag/发布。任何一步都不能借用旧版本证据，也不能由候选工作流自动授权。

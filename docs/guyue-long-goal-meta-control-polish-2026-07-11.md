# 古月 Long Goal 元控制打磨报告

日期：2026-07-11 至 2026-07-12
范围：World 长线任务证据、古月人格与路由、Long Goal v4 协议、模板、检查器、故障注入和独立只读审查
结论：升级应落在古月现有根人格与 Long Goal 能力中，不新增万能 Skill 或 Custom subagent。当前候选已从“会规划和恢复”推进到“能质疑方向、管理控制修订、保存失败证据并用真实 Git 历史封账”。

## 1. 验料结论

- 真实问题成立：World 曾出现“阶段与门禁完成，但产品仍是骨架”的伪完成，随后需要新一轮总控补救。它会在任何长线项目中复发，返工成本高。
- 稳定输入成立：模糊愿景、仓库现状、用户约束、风险、证据入口和完成定义都可被结构化收集。
- 可重复步骤成立：需求审讯、事实分层、纵向切片、风险门、执行账本、失败熔断、控制修订、证据索引和封账具有稳定顺序。
- 最小封装明确：升级 `GUYUE_PRINCIPLES.md`、根 `SKILL.md`、`requirement-analysis`、Long Goal 协议/模板与检查器；不复制 World 专用规则，不引入外部工作流框架。
- 产品钩子明确：古月不只把愿景变成可验证工程，也会在事实推翻方向时停止维护旧计划的体面，转而修订控制面。

## 2. 本地证据

对 World 近两天材料的有界复核覆盖 9 个相关用户会话、4 个审查或子任务会话和 500 条消息，未发现损坏记录。关键演化由以下提交锚定：

- `43dc627e`：重建生命世界元总控与执行包。
- `e7902480`：锚定控制包基线。
- `6bb87052`：加固控制与技术栈基线。
- `ad5c2257`：把加固规则转成控制包测试。
- `0948333e`：记录复核与技术栈决定。

从这些材料抽出的控制规律：

1. 终极愿景、当前 Goal 交付、必须靠时间验证的结果不能共用一个“完成”。
2. 已验证事实、冻结决定、待证假设、最小实验必须分栏；实验通过不能外推成产品完成。
3. 旧计划、旧绿灯和旧“完成”默认没有继承权，新总控必须声明替代关系与控制基线。
4. 先证明贯穿真实输入、机制、用户结果、失败降级和证据的纵向切片，再规模化。
5. 同一设计门三次差异化实验仍失败，应熔断到设计复核，而不是继续堆实现。
6. 总控不是上帝文件。新事实证伪方向时应暂停并产生可追溯修订，不能静默降标保护旧方案。
7. 终局证据必须绑定实现状态；文档中的提交号、哈希或“全绿”文字本身不是封账。

## 4. 最终方案

### 人格与判断

- 独立判断契约：先理解用户价值，再审查用户指定手段和古月自己的旧方案；异议必须给出事实、影响、建议和可推翻条件。
- 上浮与下钻：在终极愿景、当前 Goal、阶段和最小纵向切片之间往返，不把“全局视角”误解成更多文档或更大口号。
- 诚实完成：机器门禁只证明控制完整性和证据一致性，不替代真实用户体验、价值判断或需要时间发生的结果。

### Long Goal v4 控制契约

- 将格式版本 `4` 与 Goal 控制修订 `REV-####` 分离；修订记录追加写入，只允许一个当前 `ACTIVE` 修订。
- 非首修订必须绑定唯一、已批准的 `ACT-*`；批准动作又必须绑定新鲜、通过、退出码合格的 `FINAL` 决策证据，失败的 `ATTEMPT` 不能授权恢复。
- `FINAL` 与 `ATTEMPT` 分工：前者服务终局证明，后者保留失败和探索历史；承诺、风险门、决定和证据使用稳定 ID 双向引用。
- 同一 `REV + GATE` 下三次不同实验、不同差异说明和不同失败证据触发 `BLOCKED_DESIGN_REVIEW_REQUIRED`。恢复必须保留旧失败，新增批准修订，并在账本记录阻塞到运行的转换。
- 活跃控制文档限定为 `MASTER`、`LEDGER`、`PHASE`、`EVIDENCE_INDEX` 四类，限制数量、角色唯一性和路径真实性；归档、报告、审查与证据不冒充控制面。
- 完成时所有阶段必须为“完成”，每个承诺和风险门必须落到 `FINAL` 证据，文件哈希按当前候选重新计算。

### A/B/C Git 封账

- A：实现提交。终局证据必须声明来自 `clean@A`，不能用 `external@` 或任意脏工作树代替。
- B：证据提交，必须是 A 的直接子提交，只能更新证据、报告、审查及证据索引所需文件。
- C：封账提交，必须是 B 的直接子提交，只能修改总控与执行账本，并由 `derived@master+ledger` 推导。
- A、B、C 使用完整 Git object ID，兼容 SHA-1 与 SHA-256 仓库。
- B 之后任何提交只要触碰终局证据路径，即使随后把内容还原，也会通过历史追踪被拒绝；C 后无关提交可以存在。

## 5. 攻击、失败与回刀

早期两轮独立回放先发现并修复：

1. 阻塞状态只写在文档、没有进入合法状态机。
2. 三次失败只靠关键字和数字判断，可被 `未通过`、未声明实验或任意证据绕过。

收口阶段又进行了三轮独立只读审查：

1. 首轮发现三个有效问题：失败 `ATTEMPT` 可为恢复授权；`external@` 可绕过 `clean@A`；证据在 B 后被修改再还原可绕过净差异检查。三项均转为实现约束和反例测试。
2. 次轮发现新鲜度使用包含判断，`不新鲜` 反而会命中“新鲜”。现改为解析首个规范状态词，只允许 `新鲜`，显式拒绝 `不新鲜`、`非新鲜`、`过期`、`陈旧` 和未证明状态。
3. 末轮定向复核授权证据、封账、证据索引、失败恢复、活跃文档和对应测试，结论为 `NO_P0_P2`。它保留一个非阻塞边界：首状态词后的自然语言理由仍需人类或上层审查判断是否自相矛盾。

本地门禁也保留了真实失败：

- 首次完整套件被既有 `__pycache__` 拦截，清理生成缓存后继续，而不是忽略污染。
- 新增测试文件尚未进入真实 Git 索引时，出生证明正确拒绝把工作树当作完整候选包。为不擅自暂存用户工作，改用一次性候选索引验证精确 diff。
- 嵌套临时 Git 测试继承外层 `GIT_INDEX_FILE`，把夹具路径写入候选索引。测试现显式隔离并恢复 `GIT_INDEX_FILE`、`GIT_DIR` 和 `GIT_WORK_TREE`，故障注入不再污染被测候选。
- 生产 CLI 原先把目标仓库硬编码为古月自身根目录，函数级测试未暴露这一点。现在通过 `--repo-root` 验证外部仓库，并拒绝逃逸目标根目录的总控路径。
- 多轮 Forge 的首个 ready 包通过结构检查，却被独立语义审查判定失败：ready 证据误写为 `FINAL`、脏工作树声称 clean、风险门错绑证据、冻结决定缺审计文件。检查器和模板随后支持诚实的 `dirty@...` ATTEMPT，同时终局 FINAL 仍只接受 `clean@A`。
- 三个执行 Agent 在模拟实现阶段无文件进展并被停止。闭合决定已先写入持久决策包，因此新执行器可恢复，不需要重问用户，也没有把停滞包装成进展。

这些失败不是旁枝：它们分别验证了生成物卫生、候选包完整性和测试隔离边界。

## 6. 自动化与反例

- 生产检查器与测试夹具分文件，避免 `--self-test` 把大量构造逻辑混入运行时入口。
- 当前独立测试集包含 35 个测试，覆盖 v2/v3 兼容、v4 状态隔离、修订线性、授权恢复、失败证据、新鲜度、稳定 ID、Markdown 转义、活跃文档、阶段完成、外部目标 CLI、诚实脏 ATTEMPT 和真实 Git 封账。
- 关键反例包括：假提交号、失败授权、陈旧授权、`不新鲜`、`external@` FINAL、B 后直接变更、变更后还原、角色互换、未注册文档、重复 ID、修订分叉、批准复用和三次同质实验。
- CI、出生证明和完整安装检查都要求新的独立测试文件存在，避免只发布检查器而漏掉回归契约。
- 连续生命周期模拟在一次外部临时仓库中真实运行 CLI 和 Git，串联 ready、三失败、阻塞、批准恢复、A/B/C 封账、重启和封账后篡改拒绝。
- 安装旅程模拟用真实本地裸远程和 `file://` 克隆到空 HOME，验证安装包、首次运行、生命周期、重启身份和干净工作树；收据显式声明未验证公网和真实用户价值。

## 9. 回炉边界

- 不新增 Skill：该问题没有新的稳定用户入口，属于现有 Long Goal 和人格底盘升级。
- 不引入工作流框架：当前文件协议、Python 标准库检查器和 Git 证据足够，新增框架会提高迁移与维护成本。

- 不自动发布：本轮没有提交、推送、打 tag 或创建 Release。发布仍需精确候选、公开安装与远程门禁的新证据，以及用户明确授权。
